# covgate

[![CI](https://github.com/jesse-black/covgate/actions/workflows/ci.yml/badge.svg?branch=main)](https://github.com/jesse-black/covgate/actions/workflows/ci.yml)
[![License](https://img.shields.io/badge/License-Apache_2.0-blue.svg)](./LICENSE)

A blazing-fast, zero-dependency CI/CD quality gate that enforces code coverage strictly on your pull request diffs—built for both human developers and autonomous AI agents.

Built in Rust, `covgate` evaluates **line, branch, region, and function coverage** by parsing lossless, compiler-native coverage reports. It operates entirely locally in your CI runner or agent sandbox, preventing untested code from merging without the false positives, black-box heuristics, or SaaS costs associated with other coverage tools.

Teams do not merge a repository; they merge a diff. `covgate` makes that diff the strict unit of enforcement.

## The Global Gate Flaw

Native tools like `cargo llvm-cov`, `dotnet test`, and `vitest` are incredible at establishing a global coverage baseline. However, relying strictly on global gates in a continuous integration pipeline creates a massive blind spot: **The "Watering Down" Effect.**

If you have a large codebase with 10,000 lines of code and a global coverage of 85%, your CI gate set to 80% will easily pass. But a developer could open a pull request adding 200 lines of complex, entirely untested logic. Because the codebase is so large, those 200 untested lines might only drop the global coverage to 83.3%. 

The global gate passes. The untested code merges. Over time, this slowly bleeds your overall coverage down to the bare minimum of the global threshold. High coverage on legacy code effectively subsidizes untested new code.

## The Value of Diff Coverage Gates

A diff coverage tool solves this mathematical flaw. It isolates the exact lines modified or added in a specific Git branch and calculates coverage strictly against that changeset. 

* **The Ratchet Effect:** By setting a diff coverage gate to 80%, you enforce that *every single new PR* must meet the standard. You cannot use the high coverage of legacy code to hide weak pull requests. This guarantees your global coverage either stays flat or goes up over time.
* **Refactoring Safety:** Global gates penalize good refactoring. If you delete 500 lines of obsolete, highly-tested code, your global coverage percentage drops, potentially failing the build. Diff coverage ignores the global denominator entirely, allowing developers to clean up technical debt without fighting the CI pipeline.
* **Actionable Developer Feedback:** Failing a build because "global coverage dropped from 81.2% to 80.9%" is abstract and frustrating. Failing a build because "the 15 lines you just added in `auth.rs` lack test coverage" is actionable, immediate, and localized to the exact files the developer is thinking about.

### How `covgate` Interacts with Native Metrics
`covgate` does not replace your native test runners; it consumes their outputs. It takes the native JSON report generated by your compiler, uses a Git diff to isolate the exact lines changed in the pull request, and evaluates both line coverage and logical branch/region/function coverage strictly within that changeset.

## The "Lossless-First" Philosophy

That diff-first approach only works if the coverage data itself is trustworthy.

**Why doesn’t `covgate` support Cobertura or LCOV?**

Legacy coverage formats like Cobertura and LCOV are inherently lossy. They flatten compiler and runtime coverage models down into line-oriented XML or text formats that routinely throw away the exact information needed for precise pull-request gating. 

If you gate pull requests on lossy formats, you eventually block developers for "ghost branches" they cannot actually test, or you accidentally pass diffs where important logical paths were left untouched. 

Native formats preserve the signal that matters. In LLVM’s case, that includes **region coverage**: continuous spans of executable logic that can begin and end mid-line to reflect the actual forks the compiler sees. Region coverage is a strictly stronger metric than plain line coverage, and it is exactly the kind of signal that disappears when you force rich coverage data through lossy export formats.

`covgate` takes a strict, opinionated stance: **Parse the ground-truth coverage data directly from the producer, then apply it strictly to the diff under review.**

## The Agent Feedback Loop

AI coding agents like Codex Cloud and Google Jules are transforming development, but they expose a critical flaw in existing coverage workflows: the reliance on continuous integration pipelines for feedback.

Traditionally, agents must push their changes to remote branches and wait for the CI pipeline to execute. This creates massive inefficiencies. Some platforms cannot automatically receive PR check results, requiring a human to manually copy CI failures back into the chat. Others can ingest the results but still waste expensive compute minutes idling while waiting for the external CI pipeline to run. 

Furthermore, cloud agent sandboxes deliberately remove base branches like `main` from the checkout to ensure security and isolation. This completely breaks standard local diff-coverage tools that rely on comparing against `origin/main`.

`covgate` is designed to solve this. By natively supporting a stable, per-worktree base commit, it empowers AI agents to execute a fully local gate check *inside* the task sandbox. This creates a lightning-fast, tight feedback loop without ever pushing a commit or waiting on CI.

## Supported Ecosystems

`covgate` supports native coverage formats across several language ecosystems:

* **Rust / C / C++ / Swift (LLVM JSON):** Region-aware gating from `llvm-cov` / `cargo llvm-cov`.
* **JavaScript / TypeScript (Istanbul JSON):** Accurate line and branch gating from native JSON output.
* **C# / .NET (Coverlet JSON):** Line and branch gating from Coverlet’s native JSON.

## Installation

`covgate` is distributed as a standalone, statically linked binary. It runs instantly in your CI pipeline with zero runtime dependencies.

**Via Cargo:**

```bash
# Install covgate globally via cargo
cargo install covgate
```

## Usage

Run `covgate` in your CI pipeline after your tests generate coverage artifacts. Invoke it with either a Git base reference or a diff file.

### CLI Surface

* `--coverage-json <FILE>`: Path to native JSON coverage export
* `--base <REF>`: Git base reference to diff against
* `--diff-file <FILE>`: Precomputed unified diff file
* `--fail-under-regions <PERCENT>`: Fails if changed-region coverage is below this threshold
* `--fail-under-lines <PERCENT>`: Fails if changed-line coverage is below this threshold
* `--fail-under-branches <PERCENT>`: Fails if changed-branch coverage is below this threshold
* `--fail-under-functions <PERCENT>`: Fails if changed-function coverage is below this threshold
* `--fail-uncovered-regions <MAX>`: Fails if the raw count of uncovered regions exceeds this limit
* `--fail-uncovered-lines <MAX>`: Fails if the raw count of uncovered lines exceeds this limit
* `--fail-uncovered-branches <MAX>`: Fails if the raw count of uncovered branches exceeds this limit
* `--fail-uncovered-functions <MAX>`: Fails if the raw count of uncovered functions exceeds this limit
* `--markdown-output <FILE>`: Write a Markdown summary for CI interfaces like GitHub Actions

`covgate` also supports a dedicated agent-workflow command:

* `record-base`: Records `HEAD` into `refs/worktree/covgate/base` if that ref is not already set.

### Gating a Pull Request Locally

```bash
# Generate native JSON coverage report
cargo llvm-cov --json --output-path coverage.json

# Run covgate against the origin/main branch, failing if uncovered regions > 1
covgate --coverage-json coverage.json --base origin/main --fail-uncovered-regions 1
```

### Autonomous Agent Workflows: Recording a Stable Base

In cloud agent environments, standard base branches like `origin/main` are intentionally inaccessible for security sandboxing. Run `covgate record-base` at task start to capture a stable per-worktree base commit.

When `--base` is omitted, `covgate` automatically checks `refs/worktree/covgate/base` before the standard fallback refs used in normal local/CI workflows (`origin/HEAD`, `origin/main`, `main`). Explicit `--base` still takes precedence.

```bash
# Capture a stable base commit at task start
covgate record-base

# ...agent performs the task work...

# Generate coverage and gate locally against the recorded base
cargo llvm-cov --json --output-path coverage.json
covgate --coverage-json coverage.json --fail-under-regions 90
```

The Codex Cloud environment settings maintenance script should include `covgate record-base` so coverage gating can validate the task reliably. Jules does not have a maintenance-script setting, so `AGENTS` instructions should require running `covgate record-base` before every task.

If you prefer to run the underlying Git plumbing directly, these are the commands `covgate record-base` uses:

```bash
# Check if the base ref exists, and if not, record HEAD
git rev-parse -q --verify refs/worktree/covgate/base >/dev/null || \
  git update-ref refs/worktree/covgate/base HEAD
```

### Configuration (`covgate.toml`)

`covgate` reads repository-local defaults from `covgate.toml` at the repository root so teams can keep their gate configuration checked in with the code it protects. CLI flags always override config values.

You can now specify your default `base` and `markdown_output` at the top level, along with both minimum percentage (`fail_under_*`) and maximum uncovered count (`fail_uncovered_*`) rules under `[gates]`.

```toml
# Set a default comparison base and output file
base = "origin/main"
markdown_output = "summary.md"

[gates]
# Percentage-based gates
fail_under_regions = 90
fail_under_lines = 80
fail_under_branches = 80
fail_under_functions = 100

# Raw count gates (fail if the count is strictly greater than this value)
fail_uncovered_regions = 1
fail_uncovered_lines = 5
```

With `covgate.toml` checked in, the local invocation becomes completely frictionless:

```bash
# Run covgate using the thresholds and base defined in covgate.toml
covgate --coverage-json coverage.json
```

## GitHub Actions

Generate your native JSON coverage, run `covgate`, and seamlessly write the results to your PR summary. 

```yaml
- name: Generate Coverage
  run: cargo llvm-cov --json --output-path coverage.json

- name: Gate Pull Request
  run: covgate --coverage-json coverage.json --markdown-output "$GITHUB_STEP_SUMMARY"
```

Because `covgate` supports repository-local defaults, the checked-in `covgate.toml` guarantees your local hooks and CI pipelines are enforcing the exact same thresholds.

## How does `covgate` compare to existing tools?

There are good tools in the coverage space, but `covgate` is built with a singular focus: strict, local diff enforcement. It exists to eliminate the SaaS overhead of hosted platforms, the lossy-data compromises of legacy formats, and the CI-wait friction that cripples both human developers and autonomous agents.

**Hosted CI Platforms (SonarQube & Codecov)**
* **The Trade-off:** These are heavy, CI-bound platforms. They bring analytics layers some teams want, but require sending proprietary coverage data to a hosted system and paying SaaS fees. Crucially, they break the autonomous feedback loop by forcing developers and AI agents into a slow, asynchronous wait for external pipelines to finish.
* **The `covgate` Advantage:** `covgate` stays entirely inside the runner or agent sandbox you already trust. No hosted workflow, no SaaS costs, and an instantaneous, fully local feedback loop.

**Local Diff Tools (e.g., `diff_cover`)**
* **The Trade-off:** While tools like Python's `diff_cover` avoid the CI-wait by running locally, they assume a traditional git checkout. They lack a `record-base` equivalent, meaning they break down in cloud agent sandboxes where `main` is stripped out. Additionally, they rely on legacy formats, flattening rich coverage data down to lines and requiring a heavy language runtime.
* **The `covgate` Advantage:** `covgate` is built with first-class AI agent support via `record-base`. It ships as a single, compiled binary, reads compiler-native JSON directly, and enforces strict logical metrics (like LLVM regions and branches) without losing fidelity.

**`covrs` (Rust)**
* **The Trade-off:** Its center of gravity is reporting and SQLite aggregation rather than acting as a hard diff gate.
* **The `covgate` Advantage:** `covgate` is built from the ground up to produce an explicit pass or fail on changed code, explicitly blocking bad code from merging.
