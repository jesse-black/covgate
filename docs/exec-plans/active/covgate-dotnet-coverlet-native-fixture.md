This plan extends the completed LLVM fixture expansion work and applies the same live-fixture philosophy to .NET by introducing a real Coverlet-native JSON fixture generated via `dotnet test --collect:"XPlat Code Coverage"`.

## Purpose / Big Picture

After this change, `covgate` will support a first-class .NET fixture family that emits Coverlet-native coverage output and can be gated with the same diff-focused threshold flow already used for LLVM JSON fixtures.

Today, the repository has mature LLVM fixture coverage for Rust/C++/Swift. The next milestone is to prove that `covgate` can ingest another ecosystem’s native format, not just LLVM. The expected user-facing outcome is that a repository with .NET tests can run `covgate --coverage-json <coverlet-file>` and receive correct changed-line/branch/region gate decisions.

A second product decision is included here: whether `covgate` should continue to rely on explicit report-format switches or proactively inspect JSON payloads to autodetect parser type (LLVM JSON vs Coverlet-native JSON) so there is one stable CLI/config switch for coverage input.

## Surprises & Discoveries

- Observation: The existing fixture strategy already assumes native ecosystem tooling and checked-in artifacts, which maps cleanly to .NET if regeneration is captured via xtask rather than ad hoc shell snippets.
  Evidence: `docs/TESTING.md` requires native toolchain generation and encourages checked-in artifacts for deterministic routine test runs.

- Observation: The completed LLVM fixture plan established a reusable cross-language test harness split (`cli_metrics` vs `cli_interface`) that should be reused for Coverlet onboarding instead of adding format-specific one-off tests.
  Evidence: `docs/exec-plans/completed/covgate-cpp-swift-llvm-fixtures.md` codifies the fixture matrix and helper-module approach.

- Observation: Coverlet’s native JSON schema may not map 1:1 to LLVM “regions,” so metric availability behavior must remain explicit and tested, not assumed.
  Evidence: Existing `covgate` behavior already distinguishes unavailable metrics (for example Rust LLVM branch absence in some scenarios).

## Decision Log

- Decision: Add a real .NET fixture generated by `dotnet test --collect:"XPlat Code Coverage"` and keep the generated native JSON artifact under source control.
  Rationale: This matches the repository’s deterministic fixture policy and avoids synthetic hand-authored schema examples.
  Date/Author: 2026-03-14 / Codex

- Decision: Parser implementation should target Coverlet-native JSON directly rather than requiring conversion to Cobertura or LCOV.
  Rationale: The plan goal is first-class native-format support and parity with existing LLVM-native ingestion.
  Date/Author: 2026-03-14 / Codex

- Decision: Include parser autodetection analysis in scope and treat it as a concrete deliverable, not an optional follow-up.
  Rationale: Supporting multiple JSON formats behind one `--coverage-json` path can quickly become user-hostile if format selection UX is unclear.
  Date/Author: 2026-03-14 / Codex

- Decision: Prefer a hybrid parser-selection strategy: keep an optional explicit format override, but default to robust JSON probing.
  Rationale: Autodetect gives ergonomic “single switch” behavior, while explicit override is valuable for ambiguous or future schema drift cases.
  Date/Author: 2026-03-14 / Codex

## Outcomes & Retrospective

This section will be completed after implementation. Expected end-state outcomes:

1. `covgate` can parse and gate native Coverlet JSON in real copied-fixture CLI tests.
2. The fixture matrix includes at least one .NET pass scenario and one .NET fail scenario using changed-code diffs.
3. Parser-selection UX is documented and tested, with either deterministic autodetect or clear override semantics.
4. Existing LLVM workflows remain stable and unchanged for current users.

Likely risks to track during implementation:

- Coverlet schema variability across SDK/test collector versions could break brittle parsers.
- Path normalization for Windows-style source paths may require extra canonicalization in fixture and parser logic.
- If region/function semantics cannot be represented from native Coverlet payloads, `covgate` must report metric unavailability clearly and consistently.

## Implementation Plan

1. Add .NET fixture skeleton and baseline repository content.

   Working directory: repository root.

   Create fixture directories mirroring current conventions:

   - `tests/fixtures/dotnet/<scenario>/repo`
   - `tests/fixtures/dotnet/<scenario>/overlay`
   - `tests/fixtures/dotnet/<scenario>/coverage.json` (or similarly named native artifact)

   Expected outcome: integration harness can copy and initialize a real .NET sample repository and overlay changed files for diff tests.

2. Add xtask support to regenerate .NET native coverage artifact.

   Working directory: repository root.

   Extend xtask fixture-regeneration commands so a maintainer can run:

       cargo xtask regen-fixture-coverage dotnet/<scenario>

   The xtask should invoke native commands, including:

       dotnet test --collect:"XPlat Code Coverage"

   and copy/normalize the resulting Coverlet-native JSON into the checked-in fixture artifact path.

   Expected outcome: fixture artifact updates are reproducible and policy-compliant with `docs/TESTING.md`.

3. Implement Coverlet-native parser adapter.

   Working directory: repository root.

   Add parser module(s) under `src/coverage/` to deserialize Coverlet-native schema and convert into `covgate` model opportunities and hit counts.

   Required behaviors:

   - map line-level hits for changed-line gating
   - map branch opportunities/hits when schema provides branch detail
   - expose unavailable metrics explicitly when schema lacks equivalent fields
   - normalize file paths to match diff paths used by existing fixture harness

   Expected outcome: `covgate` can compute changed coverage metrics from Coverlet-native JSON without requiring preprocessing.

4. Implement parser selection and JSON probing strategy.

   Working directory: repository root.

   Add a format-detection layer that inspects top-level JSON shape before parser dispatch.

   Discussion and acceptance requirements:

   - Probe for LLVM-specific keys (for example `data` with LLVM export shape).
   - Probe for Coverlet-native keys (module/type/method hierarchy and line/branch structures).
   - If exactly one known format matches, dispatch automatically.
   - If none match, produce an actionable error naming supported formats.
   - If multiple match or shape is ambiguous, require/advise explicit override.
   - Preserve a single primary input switch (`--coverage-json`) and decide whether explicit format flag is retained as advanced override.

   Expected outcome: most users provide only one coverage file argument and `covgate` selects the parser safely.

5. Expand tests using the shared CLI harness pattern.

   Working directory: repository root.

   Update or add tests to follow the split from the completed LLVM plan:

   - `tests/cli_metrics.rs`: run compatible metric-threshold scenarios across LLVM and .NET fixtures.
   - `tests/cli_interface.rs`: assert parser selection UX, error messages, and config precedence.
   - shared helper module: include .NET fixture descriptors and declared metric capabilities.

   Include both positive and negative scenarios:

   - Coverlet fixture passes line threshold
   - Coverlet fixture fails line threshold
   - branch-threshold pass/fail where branch data exists
   - metric-unavailable behavior where data is absent
   - autodetect success and explicit-override fallback behavior

   Expected outcome: format support is validated at CLI level, not only unit deserialization tests.

6. Update docs and complete validation.

   Working directory: repository root.

   Update user documentation for supported coverage formats, parser selection rules, and fixture maintenance commands.

   Run full validation per repository policy:

       cargo fmt
       cargo check
       cargo clippy --all-targets --all-features -- -D warnings
       cargo test
       cargo llvm-cov --summary-only
       cargo xtask validate

   Expected outcome: repository remains green with documented and reproducible .NET coverage support.

## Validation and Acceptance

Acceptance is complete only when all of the following are true:

- A checked-in .NET fixture exists and its coverage artifact is generated by native `.NET` tooling through xtask workflow.
- `covgate` can ingest that native artifact and compute changed-file metrics without external conversion.
- CLI metric tests prove pass/fail semantics for at least line thresholds on .NET fixture(s).
- If branch data is present in fixture artifacts, branch thresholds are also validated for pass/fail; if not, explicit metric-unavailable behavior is tested.
- Parser autodetection behavior is tested and documented, including at least one unknown-format failure path.
- Existing LLVM fixture tests continue to pass unchanged.
- `cargo xtask validate` passes before completion.

## Idempotence and Recovery

- Re-running fixture regeneration for a specific .NET scenario should only update that scenario’s committed coverage artifact and leave unrelated fixtures untouched.
- Re-running integration tests should always operate on temporary copied fixture repositories and never mutate checked-in fixture source trees.
- If parser probing fails due to unexpected schema variation, recovery path is to use explicit parser override (if retained) and capture the new schema shape in tests before broadening autodetect logic.
- If `.NET` collector output path changes across SDK versions, xtask should be the sole place that resolves and normalizes those paths.
